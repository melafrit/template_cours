# I18N — Convention bilingue FR/EN

Ce document décrit, de façon opérationnelle, **comment fonctionne le bilinguisme** du template
(`site/`) et **comment l'étendre**. Tout est statique (HTML/CSS/JS vanilla), sans build, sans
dépendance réseau. Le cœur du mécanisme tient dans **un seul fichier** : `assets/js/i18n.js`,
épaulé par une règle CSS dans `assets/css/components.css`.

La langue active est portée par l'attribut `data-lang` sur `<html>` ; le contenu de l'autre
langue est **masqué en CSS** (jamais retiré du DOM) ; un bouton 🌐 injecté permet de basculer ;
le choix est mémorisé dans `localStorage`. Les moteurs JS qui génèrent de l'interface écoutent
l'évènement `tpl:langchange` pour se ré-afficher.

---

## 1. Le mécanisme

### 1.1 `data-lang` sur `<html>` + masquage CSS

`i18n.js` pose deux attributs sur la racine du document à chaque application de langue :

```js
var root = document.documentElement;
root.setAttribute('data-lang', current);  // 'fr' ou 'en' — pilote le CSS
root.setAttribute('lang', current);        // accessibilité / SEO
```

La règle qui rend tout cela visible est dans `assets/css/components.css` (section « i18n bilingue ») :

```css
html[data-lang="fr"] [data-tr="en"] { display: none !important; }
html[data-lang="en"] [data-tr="fr"] { display: none !important; }
```

**Conséquence** : on écrit le contenu **deux fois**, dans deux éléments jumeaux marqués
`data-tr="fr"` / `data-tr="en"`. Le CSS masque la version inactive. Aucun texte n'est dupliqué
en JS ni supprimé du DOM — la bascule est purement visuelle, donc instantanée et hors-ligne.

```html
<h1 data-tr="fr">Concevoir un cours en ligne</h1>
<h1 data-tr="en">Designing an online course</h1>

<p>
  <span data-tr="fr">Texte français, avec du <strong>gras</strong> autorisé.</span>
  <span data-tr="en">English text, <strong>bold</strong> allowed.</span>
</p>
```

> Les deux blocs doivent être **frères** (même parent) et porter un balisage cohérent
> (deux `<h1>`, ou deux `<span>` inline). Le HTML interne est libre.

### 1.2 Langue initiale (priorité)

Au chargement, `i18n.js` choisit `current` dans cet ordre :

1. **`localStorage['tpl-lang']`** si c'est `'fr'` ou `'en'` (choix précédent de l'utilisateur) ;
2. sinon la **langue du navigateur** (`navigator.language`) si elle est supportée ;
3. sinon l'attribut `lang` du `<html>` de la page (défaut du document), `'fr'` en dernier recours.

```js
var KEY = 'tpl-lang';
var SUPPORTED = ['fr', 'en'];
var DEFAULT = (document.documentElement.getAttribute('lang') || 'fr').slice(0, 2);
```

`apply()` est appelé **immédiatement** (avant `DOMContentLoaded`) pour éviter un *flash* de la
mauvaise langue.

### 1.3 Le bouton 🌐 injecté

Le bouton n'est **pas** écrit dans le HTML : `i18n.js` l'injecte dans `.header-right`, juste
avant `.theme-toggle` s'il existe. Il suffit donc que la page ait un conteneur `.header-right`.

```js
var hr = document.querySelector('.header-right');
var btn = document.createElement('button');
btn.className = 'lang-toggle';
btn.addEventListener('click', function () { TPLI18N.toggle(); });
```

Son libellé montre la langue **vers laquelle on bascule** (`🌐 EN` quand on est en FR), et son
`aria-label` / `aria-pressed` sont tenus à jour à chaque `apply()`. Style : `.lang-toggle` dans
`components.css`.

### 1.4 Persistance

À chaque `set()`, la langue est écrite dans `localStorage['tpl-lang']` (clé `KEY`). Les accès
sont protégés par `try/catch` (mode privé, quota), donc l'absence de `localStorage` ne casse rien.

## 2. L'API `window.TPLI18N`

`i18n.js` expose un objet global. C'est **le seul point d'entrée** ; n'écrivez jamais `data-lang`
ou `tpl-lang` à la main.

| Méthode | Effet |
|---|---|
| `TPLI18N.lang()` | Retourne la langue active : `'fr'` ou `'en'`. |
| `TPLI18N.supported()` | Retourne une copie de `['fr', 'en']`. |
| `TPLI18N.set('en')` | Force une langue. Ignore une valeur non supportée ou inchangée. Persiste, ré-applique, **émet `tpl:langchange`**. |
| `TPLI18N.toggle()` | Bascule FR↔EN (raccourci sur `set`). |
| `TPLI18N.refresh(root)` | Ré-applique **les attributs** i18n (`aria/title/alt/placeholder`) sur un sous-arbre. À appeler après une **injection HTML dynamique** portant ces attributs. |

### L'évènement `tpl:langchange`

Émis sur `document` à chaque changement effectif de langue :

```js
document.dispatchEvent(new CustomEvent('tpl:langchange', { detail: { lang: current } }));
```

Tout module qui **génère de l'UI en JS** doit s'y abonner pour se redessiner :

```js
document.addEventListener('tpl:langchange', function (e) {
  // e.detail.lang === 'fr' | 'en'
  reRender();
});
```

> Les blocs `data-tr` statiques **n'ont pas besoin** d'écouter l'évènement : le CSS s'en charge
> tout seul. L'évènement sert uniquement à l'UI générée et aux attributs ré-appliqués.

## 3. Trois manières de traduire (selon la cible)

Il y a **trois cas distincts**, à ne pas confondre.

### 3.1 CONTENU rédigé (HTML) → blocs `data-tr`

Pour tout texte visible écrit en dur dans une page. Voir §1.1. C'est le cas le plus fréquent
(titres, paragraphes, cartes, libellés de boutons-liens…). Exemple réel tiré de `index.html` :

```html
<a class="btn btn-primary" href="chapitres/01-cadrage.html">
  <span data-tr="fr">Commencer le chapitre 1 →</span>
  <span data-tr="en">Start chapter 1 →</span>
</a>
```

Aucun JS à écrire : le masquage CSS suffit.

### 3.2 ATTRIBUTS traduisibles → `data-i18n-<attr>-fr` / `-en`

Un attribut (`aria-label`, `title`, `alt`, `placeholder`) ne peut pas être « jumelé » comme un
élément. On déclare donc **les deux valeurs en attributs** et `i18n.js` **échange l'attribut réel**
au changement de langue. Quatre attributs sont gérés :

| Attribut cible | Paire à déclarer |
|---|---|
| `aria-label` | `data-i18n-aria-fr` / `data-i18n-aria-en` |
| `title` | `data-i18n-title-fr` / `data-i18n-title-en` |
| `alt` (images) | `data-i18n-alt-fr` / `data-i18n-alt-en` |
| `placeholder` | `data-i18n-ph-fr` / `data-i18n-ph-en` |

Mettez aussi une valeur **par défaut** dans l'attribut réel (pour le no-JS / le premier rendu).
Exemple réel (`glossaire.html`) :

```html
<input id="glossary-search-input"
       data-i18n-ph-fr="Rechercher un terme…"   data-i18n-ph-en="Search a term…"
       placeholder="Rechercher un terme…"
       data-i18n-aria-fr="Rechercher un terme dans le glossaire"
       data-i18n-aria-en="Search a term in the glossary"
       aria-label="Rechercher un terme dans le glossaire">
```

Et pour une image :

```html
<img src="schema.svg"
     data-i18n-alt-fr="Schéma de la progression"
     data-i18n-alt-en="Progression diagram"
     alt="Schéma de la progression">
```

Le code qui fait l'échange (`applyAttributes` dans `i18n.js`) ne sélectionne que les éléments
portant **au moins** une de ces paires :

```js
var els = root.querySelectorAll(
  '[data-i18n-aria-fr],[data-i18n-title-fr],[data-i18n-alt-fr],[data-i18n-ph-fr]');
els.forEach(function (el) {
  var a;
  if ((a = el.getAttribute('data-i18n-aria-'  + l)) != null) el.setAttribute('aria-label',  a);
  if ((a = el.getAttribute('data-i18n-title-' + l)) != null) el.setAttribute('title',       a);
  if ((a = el.getAttribute('data-i18n-alt-'   + l)) != null) el.setAttribute('alt',         a);
  if ((a = el.getAttribute('data-i18n-ph-'    + l)) != null) el.setAttribute('placeholder', a);
});
```

> **Injection dynamique** : si vous ajoutez au DOM du HTML contenant ces attributs, appelez
> `TPLI18N.refresh(monSousArbre)` ensuite, sinon ils ne seront traduits qu'au prochain
> changement de langue.

### 3.3 UI générée en JS → motif `S = {fr, en}` + `t()` + ré-affichage

Quand l'interface est **construite par du JS** (l'arbre de navigation, le QCM, les lecteurs
média, les flashcards…), il n'y a pas de double bloc HTML à masquer. Le motif standard est :

1. un dictionnaire de libellés `S` indexé par langue ;
2. un accesseur `t(k)` qui lit `S[langue]` avec repli sur `fr` ;
3. un ré-affichage sur `tpl:langchange`.

`navigation.js` est l'exemple de référence :

```js
function lang() { return window.TPLI18N ? TPLI18N.lang() : 'fr'; }

var UI = {
  fr: { chapters: 'Chapitres', resources: 'Ressources', home: 'Accueil du cours', /* … */ },
  en: { chapters: 'Chapters',  resources: 'Resources',  home: 'Course home',     /* … */ }
};
function t(k) { return (UI[lang()] || UI.fr)[k]; }

// … renderTree() utilise t('chapters'), t('resources'), etc.

renderTree();
document.addEventListener('tpl:langchange', function () { renderTree(); renderChapterDone(); });
```

Le **repli systématique sur `fr`** (`UI[lang()] || UI.fr`) évite un libellé vide si une clé
manque dans une langue. On retrouve exactement ce motif dans `flashcards.js` (objet `S`,
fonction `t`, `document.addEventListener('tpl:langchange', …)`) et dans `qcm-core.js`
(`function lng()`, dictionnaires `CBM` / `DIFF_LABEL`, helpers `certLabel`, `diffLabel`).

## 4. La forme bilingue des DONNÉES

Les fichiers de données ne contiennent **pas** de HTML jumelé : ils portent des **objets
`{ fr, en }`** que les moteurs résolvent à l'affichage avec un petit helper `L()` (« localize ») :

```js
function lang() { return window.TPLI18N ? TPLI18N.lang() : 'fr'; }
// L(x) : x peut être une chaîne OU un objet { fr, en }
function L(x) {
  return (x && typeof x === 'object') ? (x[lang()] || x.fr || x.en || '') : (x || '');
}
```

Ce `L()` est présent à l'identique dans `navigation.js` et `flashcards.js`. Il **tolère une
chaîne simple** (utile si vous mono-linguez une donnée) et **replie** `fr` → `en` → `''`.

### Récapitulatif par fichier de données

| Fichier | Champs bilingues `{fr,en}` | Champs non traduits |
|---|---|---|
| `assets/js/course-data.js` | `title`, `subtitle`, `license`, `chapters[].title`, `resources[].label` | `author`, `chapters[].num/ico/file`, `resources[].ico/href` |
| `assets/js/nav-sections.js` | `RDC_SECTIONS[ch][].t` | `id` (ancre `#chN-sM`) |
| `assets/js/glossaire-data.js` | `def`, `ex` (chacun `{fr,en}`) | `fr` / `en` = **le terme** lui-même dans chaque langue |
| `assets/js/flashcards-data.js` | `front`, `back` (chacun `{fr,en}`) | `ref` (chaîne, ex. `"art. 5"`) |
| `QCM/questions.js` | `enonce`, chaque `propositions[]`, `meta.titre`, `meta.formation` | `id`, `chapitre`, `difficulte`, `meta.auteur`, `meta.annee_academique` |
| `QCM/corrections.js` | `explication` | `id`, `chapitre`, `bonne_reponse_index` |

### Exemples exacts

**`course-data.js`** — titres/libellés bilingues, identifiants techniques en chaîne simple :

```js
window.COURSE = {
  title:    { fr: "Concevoir un cours en ligne", en: "Designing an online course" },
  author:   "Mohamed EL AFRIT",
  chapters: [
    { num: '1', ico: '🎯', file: '01-cadrage.html',
      title: { fr: "Cadrer son cours", en: "Scoping your course" } }
  ],
  resources: [
    { ico: '📚', href: 'glossaire.html', label: { fr: "Glossaire bilingue", en: "Bilingual glossary" } }
  ]
};
```

**`glossaire-data.js`** — `fr`/`en` portent **le terme**, `def`/`ex` sont eux-mêmes `{fr,en}` :

```js
{ fr: "objectif pédagogique", en: "learning objective",
  def: { fr: "Ce que l'apprenant doit être capable de FAIRE à la fin…",
         en: "What the learner should be able to DO at the end…" },
  ex:  { fr: "« À la fin du chapitre, l'apprenant rédige… »",
         en: "\"By the end of the chapter, the learner writes…\"" } }
```

**`flashcards-data.js`** — `front`/`back` bilingues (le `back` accepte `**gras**`), `ref` neutre :

```js
{ front: { fr: "Qu'est-ce qu'un objectif pédagogique ?", en: "What is a learning objective?" },
  back:  { fr: "Ce que l'apprenant doit savoir **faire** à la fin…",
           en: "What the learner must be able to **do** at the end…" },
  ref: "" }
```

> Le `**gras**` du `back` est interprété par `flashcards.js` (`rich()`), pas par le navigateur :
> le texte est échappé puis seuls les `**…**` deviennent `<strong>`. N'y mettez pas de HTML.

**`QCM/questions.js`** + **`QCM/corrections.js`** — appariés par `id` ; la bonne réponse est un
index, l'explication est bilingue :

```js
// questions.js
{ id: 'Q01', chapitre: 1, difficulte: 'vert',
  enonce: { fr: 'Quel est le tout premier élément à définir…',
            en: 'What is the very first element to define…' },
  propositions: [
    { fr: 'Les objectifs pédagogiques visés', en: 'The intended learning objectives' },
    { fr: 'La couleur du thème graphique',     en: 'The colour of the visual theme' }
  ] }

// corrections.js (même id)
{ id: 'Q01', chapitre: 1, difficulte: 'vert', bonne_reponse_index: 0,
  explication: { fr: 'On part toujours des objectifs pédagogiques…',
                 en: 'You always start from the learning objectives…' } }
```

**`nav-sections.js`** — sections de chapitre, `id` = ancre, `t` = titre bilingue :

```js
window.RDC_SECTIONS = {
  "1": [ { id: "ch1-s1", t: { fr: "1. Pourquoi cadrer avant de produire",
                              en: "1. Why scope before producing" } } ]
};
```

### Règle d'or pour ajouter une donnée

Tout champ **affiché à l'utilisateur** doit être un objet `{ fr, en }` (les deux clés présentes).
Tout champ **technique** (id, fichier, index, ancre, icône, href) reste une **chaîne neutre**.
En cas de doute : si le texte change quand on bascule la langue, il est bilingue.

## 5. Faire varier le nombre de langues

### 5.1 Passer le cours en UNE seule langue

Le template reste bilingue dans le code, mais vous pouvez n'exposer qu'une langue :

- **Verrouiller la langue et masquer le bouton.** Dans une feuille de style (ou à la fin de
  `components.css`), forcer FR et cacher le sélecteur :

  ```css
  .lang-toggle { display: none !important; }
  ```

  Puis, une fois par session, fixer la langue : `TPLI18N.set('fr');` au chargement (par exemple
  dans un petit script de page). Comme `set` persiste dans `localStorage`, le choix tient.

- **Côté contenu**, deux options :
  - *Le plus simple* : **ne rien supprimer**. Gardez les blocs `data-tr="fr"` et `data-tr="en"` ;
    seul le bouton disparaît, l'autre langue reste masquée par le CSS. Zéro réécriture.
  - *Allègement* : ne garder que les blocs de la langue retenue et retirer les jumeaux. Pour les
    **données**, `L()` accepte une **chaîne simple** — vous pouvez donc écrire
    `title: "Cadrer son cours"` au lieu de `{ fr, en }` sans rien casser. Pour la **UI générée**,
    laissez les dictionnaires `S/UI` tels quels (le repli sur `fr` les rend inoffensifs).

> Recommandation : préférez le **masquage** (CSS + bouton caché) à la suppression. Vous restez
> réversible et vous ne touchez ni au HTML ni aux données.

### 5.2 Ajouter une 3e langue (ex. `es`) — en théorie

Le mécanisme est conçu pour 2 langues mais reste extensible. Il faut intervenir aux **cinq**
mêmes endroits que le bilinguisme :

1. **`i18n.js` — déclarer la langue.** Ajouter `'es'` à `SUPPORTED` :

   ```js
   var SUPPORTED = ['fr', 'en', 'es'];
   ```

   Adapter le **bouton** : il ne peut plus afficher « l'autre » langue, il doit **cycler**.
   Remplacer la logique du libellé (qui code en dur `current === 'fr' ? 'EN' : 'FR'`) par un
   cycle sur `SUPPORTED`, et faire de même pour `toggle()` (passer à la langue suivante de la
   liste plutôt qu'un simple FR↔EN). C'est la seule vraie modification de logique.

2. **CSS — masquer les langues inactives.** Ajouter les règles pour `es` :

   ```css
   html[data-lang="fr"] [data-tr="es"],
   html[data-lang="en"] [data-tr="es"] { display: none !important; }
   html[data-lang="es"] [data-tr="fr"],
   html[data-lang="es"] [data-tr="en"] { display: none !important; }
   ```

   (Plus maintenable : « masquer tout `[data-tr]` dont la valeur n'est pas la langue active » —
   mais le CSS ne sait pas comparer dynamiquement, donc il faut énumérer les paires.)

3. **CONTENU HTML** — ajouter un troisième bloc `data-tr="es"` à côté de chaque paire FR/EN.

4. **ATTRIBUTS** — ajouter les variantes `data-i18n-aria-es`, `-title-es`, `-alt-es`, `-ph-es`.
   `applyAttributes` lit déjà `'data-i18n-aria-' + l` : avec `l === 'es'`, **aucun changement**
   n'est requis dans `i18n.js` pour les attributs (juste les déclarer dans le HTML).

5. **UI générée + DONNÉES** — ajouter la clé `es` partout :
   - dictionnaires `S` / `UI` / `CBM` / `DIFF_LABEL` : ajouter un bloc `es: { … }` ;
   - le helper `L()` replie déjà `x[lang()] || x.fr || x.en` — il **fonctionne** avec `es`, mais
     prévoyez `es` dans les objets de données (`{ fr, en, es }`) sinon il retombera sur `fr` ;
   - `t()` / `certLabel()` etc. : le repli `|| S.fr` couvre les clés manquantes.

> En résumé : **le seul code à réécrire** est la logique du bouton (libellé + `toggle`) dans
> `i18n.js` ; tout le reste est de la donnée à compléter. Le repli sur `fr` partout garantit
> qu'une traduction `es` incomplète dégrade proprement au lieu de casser l'affichage.

## Checklist — ajouter une page / un composant bilingue

- [ ] La page contient `.header-right` (le bouton 🌐 s'y injecte tout seul).
- [ ] Charger `assets/js/i18n.js` (idéalement tôt) ; le CSS `components.css` est inclus.
- [ ] Tout texte en dur est en **double bloc** `data-tr="fr"` / `data-tr="en"` (frères).
- [ ] Les `aria-label` / `title` / `alt` / `placeholder` utilisent les paires `data-i18n-*-fr/-en`
      **et** une valeur par défaut dans l'attribut réel.
- [ ] Toute UI générée en JS suit le motif `S={fr,en}` + `t()` + écoute de `tpl:langchange`.
- [ ] Les données ajoutées sont des objets `{ fr, en }` (champs visibles) ; champs techniques neutres.
- [ ] Après une **injection HTML dynamique** porteuse d'attributs i18n : appeler `TPLI18N.refresh(root)`.
- [ ] Ne jamais écrire `data-lang` / `tpl-lang` à la main : passer par `TPLI18N.set` / `toggle`.
